# IRMS Phase 1 (Core) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** A working web app where a COMSATS Sahiwal CS student signs up with their university email (OTP-verified), submits a reference-letter request, a coordinator approves it, and a professional PDF with an auto serial number and a public QR-verification page is generated for download.

**Architecture:** Single Next.js (App Router) application, deployed on Vercel, backed by Supabase (Postgres + Auth + Storage). Brevo provides SMTP for OTP/email. The UI branches by role (student/coordinator/hod/admin) after login. PDFs are generated server-side and stored in Supabase Storage; each letter carries an unguessable token used by a public verification page.

**Tech Stack:** Next.js 15 (App Router, TypeScript), Tailwind CSS, Supabase (`@supabase/supabase-js`, `@supabase/ssr`), pdf-lib, qrcode, Vitest + Testing Library for unit tests.

## Global Constraints

- **Zero cost:** only free tiers (Supabase free, Vercel hobby, Brevo free). No paid services. No SMS, no AI in Phase 1.
- **Student email domain:** signup allowed only for emails ending in `@students.cuisahiwal.edu.pk`.
- **Registration number** is derived from the email local part `fa24-bse-011` → `CIIT/FA24-BSE-011/SWL` (campus suffix `SWL` constant). Session=`FA24`, program=`BSE`/`BCS`, roll=`011`.
- **Programs supported:** `BCS → BS Computer Science`, `BSE → BS Software Engineering`. Reject other program codes at signup.
- **Department:** `CS` (constant for now; keep the field for future).
- **Serial number format:** `IRMS-<YEAR>-<6-digit-zero-padded>`, e.g. `IRMS-2026-000001`. Must be globally unique via atomic DB increment.
- **Roles:** `student`, `coordinator`, `hod`, `admin`. Flexible approval: coordinator is primary; hod/admin can override.
- **Frontend quality:** distinctive, professional, intentional design — NOT a generic templated/AI look. Use the `frontend-design` skill when building UI. See spec §11.
- **Privacy on public verify page:** show only name, reg#, program, semester, company, issue date, status. Never phone/CNIC.
- **Secrets** live only in `.env.local` (never committed). `.env.example` documents required keys.

```ts
export const STUDENT_EMAIL_DOMAIN = 'students.cuisahiwal.edu.pk'
export const PROGRAMS = {
  BCS: 'BS Computer Science',
  BSE: 'BS Software Engineering',
} as const
export type ProgramCode = keyof typeof PROGRAMS

export function isStudentEmail(email: string): boolean {
  return email.trim().toLowerCase().endsWith('@' + STUDENT_EMAIL_DOMAIN)
}

export function parseStudentEmail(email: string) {
  const e = email.trim().toLowerCase()
  if (!isStudentEmail(e)) return null
  const local = e.split('@')[0]
  const m = local.match(/^([a-z]{2}\d{2})-([a-z]{3})-(\d{1,4})$/)
  if (!m) return null
  const session = m[1].toUpperCase()
  const program = m[2].toUpperCase()
  const roll = m[3]
  if (program !== 'BCS' && program !== 'BSE') return null
  return { session, program: program as ProgramCode, roll }
}

export function buildRegNumber(p: { session: string; program: string; roll: string }) {
  return `CIIT/${p.session}-${p.program}-${p.roll}/SWL`
}
```

```sql
-- Atomically returns the next serial counter for a given year.
create or replace function next_serial(p_year int)
returns int language plpgsql as $$
declare v int;
begin
  insert into app_settings(key, value)
  values ('serial_counter_' || p_year, '1')
  on conflict (key) do update
    set value = (app_settings.value::int + 1)::text
  returning value::int into v;
  return v;
end; $$;
```
